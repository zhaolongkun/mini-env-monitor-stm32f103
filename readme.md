# STM32F103C8T6 迷你环境监测仪设计与思考文档

## 1. 项目简介

本项目基于 STM32F103C8T6 最小系统板实现一个迷你环境监测仪。系统通过 DHT11 温湿度传感器采集环境温度和湿度，通过 I2C 接口驱动 SSD1306 OLED 显示屏实时显示温度、湿度和系统运行时间。同时，系统使用一个独立按键实现显示页面切换，使用 PC13 板载 LED 作为状态指示灯，并通过 USART1 串口以 JSON 格式周期性上报温湿度数据。

本项目采用 STM32CubeMX + HAL 库开发，代码采用分层架构设计，主要分为 Drivers、App、Utils 三层，便于模块复用、功能扩展和后续维护。

## 4. 模块划分说明

### 4.1 Drivers 层

Drivers 层负责具体外设驱动，屏蔽底层硬件细节。

* `Drivers/DHT`：负责 DHT11 温湿度传感器的数据读取。
* `Drivers/OLED`：负责 SSD1306 OLED 初始化、清屏和字符串显示。
* `Drivers/Button`：负责按键扫描、消抖和短按事件检测。
* `Drivers/LED`：负责 LED 开、关、翻转，并封装 PC13 低电平点亮的硬件差异。

### 4.2 App 层

App 层负责系统应用逻辑，主要包括：

* 周期读取温湿度数据；
* OLED 页面显示；
* 按键短按切换页面；
* 温度超过 30℃ 时触发 LED 常亮报警；
* 正常状态下 LED 每秒闪烁一次；
* 每秒通过 USART1 上报 JSON 格式数据。

### 4.3 Utils 层

Utils 层负责通用工具函数。

* `filter.c/h`：实现滑动平均滤波，降低温湿度数据抖动。
* `uart_json.c/h`：负责生成 JSON 字符串并通过 UART 发送。

## 6. 非阻塞任务调度设计

为了避免主循环被某个任务长时间占用，本项目没有在 `while(1)` 中大量使用 `HAL_Delay()`，而是使用 `HAL_GetTick()` 实现非阻塞任务调度。

主要任务周期如下：

| 任务           | 周期              |
| ------------ | --------------- |
| 按键扫描         | 10 ms           |
| OLED 刷新      | 200 ms 或 500 ms |
| DHT11 温湿度读取  | 1000 ms         |
| UART JSON 上报 | 1000 ms         |
| LED 状态更新     | 1000 ms         |

这种设计方式可以保证按键检测、OLED 刷新、串口发送和 LED 控制互不阻塞，提高系统响应速度和稳定性。

## 7. 按键消抖设计

按键采用 PA0 输入上拉方式，按下时为低电平。由于机械按键在按下和松开瞬间会产生电平抖动，因此软件中采用约 20 ms 的消抖时间。

消抖逻辑如下：

1. 周期性读取按键电平；
2. 如果检测到电平变化，记录变化时间；
3. 只有当电平稳定超过 20 ms 后，才确认按键状态改变；
4. 每次短按只产生一次页面切换事件，避免一次按下触发多次切换。

## 8. 滑动平均滤波设计

DHT11 采集到的温湿度数据可能存在轻微抖动，因此本项目对温度和湿度分别使用滑动平均滤波。

滤波窗口大小设置为 5，即保存最近 5 次有效采样值，并计算平均值作为最终显示和上报数据。

这样可以降低单次异常数据对显示和串口上报结果的影响，提高系统稳定性。

当 DHT11 读取失败时，系统不会将温湿度强制置零，而是保留上一次有效值，并记录错误次数。

## 9. 三处对 AI 生成代码的修改或优化点及原因

### 9.1 将阻塞式主循环改为非阻塞任务调度

AI 初始生成代码时，容易直接在主循环中使用 `HAL_Delay()` 实现周期任务。这样虽然简单，但会导致按键响应变慢，OLED 刷新和串口发送也容易互相影响。

因此我将主逻辑改为基于 `HAL_GetTick()` 的非阻塞调度方式，不同任务按照不同周期独立执行，提高了系统实时性和稳定性。

### 9.2 DHT 读取失败时保留上一次有效数据

如果 DHT11 某次读取失败，直接把温度和湿度清零，会导致 OLED 显示和串口 JSON 数据突然跳变，不符合实际环境变化规律。

因此我优化为：当 DHT 读取失败时，不更新滤波器输入，保留上一次有效温湿度数据，同时增加错误计数。这样系统显示更加稳定，也方便在系统信息页面观察传感器通信状态。

### 9.3 封装 PC13 LED 低电平点亮逻辑

STM32F103C8T6 最小系统板的 PC13 板载 LED 通常是低电平点亮、高电平熄灭。如果在 App 层直接写 GPIO 电平，容易造成逻辑混乱。

因此我在 LED 驱动层封装 `LED_On()`、`LED_Off()` 和 `LED_Toggle()` 接口，使 App 层只关心“亮”和“灭”，不需要关心具体电平是否反向，提高了代码可读性和可移植性。

## 10. 开发过程中遇到的关键问题及解决思路

### 问题 1：DHT11 单总线时序严格

DHT11 对起始信号、响应信号和数据位读取时序要求较高，普通毫秒级延时无法满足要求。

解决思路：使用 TIM2 配置为 1 us 计数周期，在 DHT 驱动内部实现微秒级延时函数，用于控制 DHT11 通信时序。

### 问题 2：OLED I2C 地址可能不同

常见 SSD1306 OLED 模块地址一般为 0x3C，但也可能是 0x3D。如果地址错误，程序可以正常运行，但 OLED 不显示。

解决思路：将 OLED 地址集中定义在 `oled.h` 中，默认使用 0x3C。如果 OLED 不显示，可以快速修改为 0x3D 重新编译验证。

## 11. JSON 数据上报格式

系统每秒通过 USART1 上报一次 JSON 数据，格式如下：

```json
{"temp":25.5,"hum":60.0,"time":120,"alarm":0}
```

字段说明：

| 字段    | 含义                       |
| ----- | ------------------------ |
| temp  | 当前温度，单位 ℃                |
| hum   | 当前湿度，单位 %                |
| time  | 系统运行时间，单位 s              |
| alarm | 报警状态，0 表示正常，1 表示温度超过 30℃ |

## 12. 总结

本项目完成了基于 STM32F103C8T6 最小系统板的迷你环境监测仪设计，实现了温湿度采集、OLED 显示、按键页面切换、LED 状态报警、UART JSON 数据上报、按键消抖和滑动平均滤波等功能。

工程采用分层架构设计，主循环使用非阻塞任务调度方式，代码结构清晰，便于后续扩展 DHT22、更多传感器或其他通信方式。
